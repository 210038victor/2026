# 2026 專案憲法

## 核心原則

### I. 繁體中文優先 (Traditional Chinese First)

**規範**：所有規格文件、計畫文件、任務清單、以及與使用者的互動回應，**必須**使用繁體中文撰寫。

**理由**：確保專案文件的一致性與可讀性，符合目標使用者的語言偏好，降低溝通成本與誤解風險。

**適用範圍**：
- 所有 `.specify/` 目錄下的文件（spec.md、plan.md、tasks.md 等）
- 所有 AI 代理回應
- 程式碼註解（建議使用繁體中文，但不強制）
- README 與文檔

**例外**：程式碼本身（變數名、函數名、類別名）使用英文，以符合程式語言慣例。

---

### II. 簡約設計 (Simplicity First - NON-NEGOTIABLE)

**規範**：**禁止**過度設計。所有解決方案必須從最簡單的可行方案開始，只有在明確需求驅動下才增加複雜度。

**理由**：過度設計會增加維護成本、降低開發速度，並引入不必要的錯誤。YAGNI（You Aren't Gonna Need It）原則是提高生產力的關鍵。

**實踐準則**：
- 優先選擇標準函式庫而非第三方套件
- 避免引入未來「可能」需要的功能
- 每個新增的複雜度必須有明確的使用場景驗證
- 程式碼審查時質疑所有非必要的抽象層

**違反時的處理**：若發現過度設計，必須在計劃文件的「複雜度追蹤」(Complexity Tracking) 區段中提供正當理由，或進行重構簡化。

---

### III. 精簡文檔 (Minimal Documentation)

**規範**：**除非使用者明確要求**，否則**不得**建立新的 Markdown 文件來記錄變更、總結工作或產生額外的報告。

**理由**：過多的文檔會造成資訊冗餘、增加維護負擔，並分散開發者注意力。專案應依賴現有的規格文件（spec.md）、計畫文件（plan.md）和任務清單（tasks.md）。

**實踐準則**：
- 變更記錄應透過 Git 提交訊息管理
- 工作進度透過 tasks.md 的打勾機制追蹤
- 設計決策記錄在 plan.md 中
- 僅在使用者明確要求時才建立額外文檔（如 CHANGELOG.md、API 文件等）

---

### IV. Git 流程紀律 (Git Discipline - NON-NEGOTIABLE)

**規範**：每個開發階段**必須**確保 Git 正確運行，並在關鍵里程碑進行提交。

**理由**：Git 是專案歷史的唯一真相來源。缺乏適當的版本控制會導致工作遺失、難以回溯問題，並增加協作困難。

**實踐準則**：
- 每個功能分支必須從最新的主分支建立
- 完成規格階段後提交（spec.md）
- 完成計畫階段後提交（plan.md、research.md 等）
- 完成任務階段後提交（tasks.md）
- 實施階段每完成一個任務群組（Phase）後提交
- 提交訊息必須清晰描述變更內容（建議使用 Conventional Commits 格式）

**驗證點**：
- 每個階段開始前確認工作目錄乾淨或已提交
- 每個階段結束後確認新增檔案已追蹤並提交
- 使用 `git status` 確認無未追蹤或遺漏的檔案

---

### V. 測試驅動開發 (Test-Driven Development - NON-NEGOTIABLE)

**規範**：所有功能實作**必須**遵循 TDD（測試驅動開發）流程：撰寫測試 → 使用者核准 → 測試失敗 → 實作程式碼 → 測試通過 → 重構。

**理由**：TDD 確保程式碼符合需求、可測試性高、易於重構，並建立了回歸測試的安全網。測試先行也強迫開發者思考介面設計與使用者場景。

**實踐準則**：
- 在 tasks.md 中，測試任務**必須**排在實作任務之前
- 測試必須先執行並確認失敗（Red）
- 實作最小可行程式碼使測試通過（Green）
- 在確保測試通過的前提下重構（Refactor）
- 每個使用者故事必須有對應的驗收測試

**測試類型**：
- 合約測試 (Contract Tests)：驗證 API 或模組介面
- 整合測試 (Integration Tests)：驗證元件間互動
- 單元測試 (Unit Tests)：驗證個別函數或類別

---

### VI. 任務追蹤 (Task Tracking - NON-NEGOTIABLE)

**規範**：在實施 (implement) 階段，**必須**確保 tasks.md 中的任務在完成後被正確標記（打勾）。

**理由**：任務清單是進度追蹤的唯一真相來源。未正確更新會導致進度不透明、重複工作，或遺漏未完成的任務。

**實踐準則**：
- 每完成一個任務，立即在 tasks.md 中將 `[ ]` 改為 `[x]`
- 提交程式碼時，同時提交更新後的 tasks.md
- 定期檢視 tasks.md 確保所有已完成的工作都已標記
- 若發現新增的必要任務，先更新 tasks.md 再執行

**驗證點**：
- 每個實施階段的提交必須包含 tasks.md 的更新
- 階段性檢查點時驗證已打勾的任務確實已完成
- 未完成的任務不應被標記為完成

---

### VII. 規格保護 (Specification Protection - NON-NEGOTIABLE)

**規範**：在實施 (implement) 階段，**必須**確認不會意外刪除或覆蓋規格文件，特別是在套用框架模板時。

**理由**：規格文件（spec.md、plan.md 等）是功能需求與設計決策的唯一文件化記錄。遺失這些文件會導致專案失去脈絡、無法驗證實作是否符合需求。

**實踐準則**：
- 套用框架模板（如 create-react-app、create-next-app 等）前，先備份 `/specs` 目錄
- 框架初始化完成後，立即驗證 `/specs` 目錄完整性
- 在專案根目錄執行框架初始化時，使用 `--no-git` 或類似選項避免覆蓋 `.gitignore`
- 若必須在子目錄初始化框架，確保不影響專案根目錄的 `/specs` 和 `.specify`

**驗證點**：
- 框架模板套用前：`ls -la specs/` 記錄現有檔案
- 框架模板套用後：驗證 `specs/` 目錄內容未被修改
- 若發現檔案遺失，立即從 Git 復原

**特別注意的框架**：
- Next.js (`npx create-next-app`)
- React (`npx create-react-app`)
- Vite (`npm create vite@latest`)
- 任何會在專案根目錄建立大量檔案的腳手架工具

---

### VIII. 靜態網站優先 (Static Website First)

**規範**：當專案類型為網站時，**應優先**採用前端靜態網站架構，使其能夠部署於 GitHub Pages 或類似的靜態託管服務。

**理由**：靜態網站具有部署簡單、維護成本低、效能高、安全性佳等優勢。符合簡約設計原則，並降低基礎建設複雜度。

**實踐準則**：
- 優先採用靜態網站生成器（如 Next.js Static Export、Vite、Hugo 等）
- 避免引入需要伺服器端執行的功能，除非需求明確
- 若需要動態功能，優先考慮前端解決方案（如 Client-side API 呼叫、Serverless Functions）
- 設計時考慮 GitHub Pages 的部署限制（無伺服器端程式碼、無資料庫）

**適用情境**：
- 文件網站、部落格、作品集網站
- 行銷頁面、產品展示頁面
- 單頁應用程式 (SPA)
- 工具或計算機類的網頁應用

**例外情境**（需在 plan.md 中說明）：
- 需要即時協作功能（如即時聊天、協同編輯）
- 需要複雜的後端邏輯或資料處理
- 需要使用者驗證與權限管理（超出前端能力範圍）

## 開發工作流程 (Development Workflow)

### 功能開發循環

1. **規格階段 (Specify)**: 使用 `/speckit.specify` 建立或更新 spec.md
2. **釐清階段 (Clarify)**: 使用 `/speckit.clarify` 識別並解決未明確的需求
3. **計畫階段 (Plan)**: 使用 `/speckit.plan` 產生實作計畫與設計文件
4. **任務階段 (Tasks)**: 使用 `/speckit.tasks` 產生可執行的任務清單
5. **分析階段 (Analyze)**: 使用 `/speckit.analyze` 檢查各文件間的一致性
6. **實施階段 (Implement)**: 使用 `/speckit.implement` 執行任務並實作功能

### 品質門檻 (Quality Gates)

每個階段結束前必須滿足：

- **Git 提交**：所有變更已提交並有清晰的提交訊息
- **檔案完整性**：所有必要的文件已建立且格式正確
- **原則符合性**：沒有違反憲法原則，或已在計畫文件中提供正當理由
- **可測試性**：（實施階段）所有測試必須先撰寫並執行

## 治理 (Governance)

### 憲法優先級

本憲法凌駕於所有其他開發實踐之上。當憲法與其他文件或慣例衝突時，以憲法為準。

### 修正程序

修改本憲法需要：

1. **提出修正提案**：說明修正理由、影響範圍、替代方案評估
2. **影響分析**：評估對現有專案、模板、工作流程的影響
3. **版本更新**：依語義化版本規則更新版本號
   - **MAJOR**：移除或重新定義核心原則，導致向後不相容
   - **MINOR**：新增原則或章節，擴充治理範圍
   - **PATCH**：澄清措辭、修正錯字、非語義性改進
4. **同步更新**：更新所有受影響的模板與文件
5. **移轉計畫**（若需要）：對既有專案提供移轉指引

### 合規性驗證

- 所有 Pull Request 審查時必須驗證憲法合規性
- 在 plan.md 的「Constitution Check」區段檢查原則符合性
- 複雜度或例外情況必須在計畫文件中提供正當理由
- 定期審查專案是否遵循憲法（建議每個功能里程碑）

### 參考文件

- 執行指引：參考 `.github/skills/` 目錄下的各項技能檔案（SKILL.md）
- 模板文件：參考 `.specify/templates/` 目錄下的模板檔案
- 專案記憶：參考 `.specify/memory/` 目錄下的歷史記錄
